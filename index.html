为36kr的注册登录一体化点赞

發表於 2015-08-14   |   分類於 http   |  

最近公司要做一套用户注册登录系统找了很多用户系统参考后来发现了36氪的注册登录页面流程简单易用也很合乎我的想法.

基于手机号码注册登录一体化

现在的移动互联网时代手机号码注册早已经是用户习惯又愿意做的事情,而以前基于邮箱注册手段已经慢慢被淡化.

首先,注册邮箱的成本几乎为0导致邮箱账号过于泛滥,因此系统充斥着很多垃圾用户,这是投资者和管理者都不愿意看到的事情.相对手机号注册用户的价格大得多了,一般用户个人持有的手机号码最多也就3个,而不停换手机注册的成本比较高.

其次,邮件的发送成本并不低.众所周知邮箱的入口早给163 sina qq等大牌占领并牢牢控制着,邮件稍微多发邮件很容易被邮件服务器误以为是垃圾邮件,而用户收到邮件愿意看邮件的转化率相低况且现在都流行微信push的方式.手机的发送虽然需要每条5-8分钱的成本,但如果您的项目不是自己一个人小打小闹的话这个成本其实也不是很高.

所以公司的注册流程决定废除邮箱注册采用手机外加第三方登录.

流程上的减法

观看很多网站注册登录一般都需要两个按钮分别连接登录注册两个页面,而参考36氪的做法则是是统一在一起并且废除注册密码时候的密码二次输入.这样令到注册登录融为一体更加自然,因为注册和登录差距多输入一个动态验证码的过程.

  1. 如果是手机用户,输入完手机之后系统会ajax查询下改手机号码是否为已经注册的用户,如果是已注册的用户提交按钮马上变为’登录’字眼.
  2. 如果手机号码未被注册,页面会多显示一个短信动态验证码的输入框,同时提交按钮马上变成’注册’字眼.其实注册比登录就多了一个获取动态验证码的流程.
  3. 减掉密码二次确认的输入框,想必大家都习惯注册账号的时候为什么需要输入2次密码.我自己感觉这个操作流程是有点low的做法了.
    3.1 第三方登录的快捷便利减弱了传统网站的注册模式,微信,微博,qq第三账号进行直接登录不单用户操作极为简单.基本上连密码都不用输入就成为某个网站的注册用户了.那在那么便捷且更安全的情况下就可以登录了用户不乐意呢. ps:提到安全这里要多插一句,如果用户还是走原来填写自己的账号名密码方式注册其实会多增加一次账号被盗用的风险这就不展开说明了.
    3.2 基于手机注册找回密码极为方便且相对安全,密码就算忘记找回也极为方便的事. 相对于邮箱找回密码而言手机找回密码要安全性更高,因为邮箱也是有需要密码登录的也存在邮箱密码忘记和被盗的可能,而手机它就在你身边(这不讨论手机丢了的情况)
    一般用户输入密码的时候都会用自己常用的账号和密码,输入错误的情况极少.还有一种是随机生成密码这部分用户大部分都是程序员^_^二次输入只是copy2次而已其实没什么卵用.那用户二次输入密码的必要性已经不重要了.
    经过减掉注册流程密码需要二次输入流程后,其实注册流程就比登录流程多了个获取并输入动态验证码的流程.这种设计下才让注册登录融合变得那么的自然及简单其实某种程度上其实也是方便了用户.

提高网络通信的安全性

这里不详细叙述网络安全性的重要性及必要性也不详细叙述怎样实现安全性.不同的项目不同侧重点,或者很多项目只是做功能体验为主安全性对于他们来说就是个用户看不见的东西.
但公司的业务范围老板考虑的是用户资料的安全性.所以注册登录流程上我们都要考虑.
这里简单描述下某些别有用心的人怎样获取到你的账号密码,这里只举例一个例子(说的不好不要喷我也是听说的或者提出您更牛x的方法).
首先,某些人需要知道您的账号密码他们必须有个拦截点,例如:wifi.那他们怎样做到的呢?假设你的wifi 信号是ABCDE, 某些人就会用一个信号更强大的路由器信号源也是命名为ABCDE而且不需要密码
相同情况下信号强的wifi信息就会逼退信号弱的.所以你可能会突然掉下线重连.其实这时你已经连接某些人为你准备的wifi了.之后某些人就可以通过路由器抓包截获你的访问的所有通信.目前来说大部分都是走http的(明文)
你输入什么接受什么都某些人都看得一清二楚. ^-^
这种情况解决办法:
你访问的网站是走https,https是双向通讯都进行加解密,这样计算某些人截获到也没什么卵用.但走https网站服务商每年都需要给几千的费用已经忘记了是人民币还是刀了,但由于双向都要加解密访问过程就会变慢.
还有一种就是网站为用户进行加密通讯.例如: RSA. 这里就不展开了. 我们的注册登录提交都是经过RSA的.这样对重要的信息进行RSA加密,这样某种程度是速度和安全性的解决方案.

使用Git来管理hexo的博客源文件

發表於 2015-08-14   |   分類於 hexo   |  

之前用MacBook air使用hexo写博客的一个问题就是源文件都是在本地的,公司配置了MacBook pro就出现公司一个hexo,家里的hexo不能同步的问题
觉得比较靠谱的办法还是用git来管理了。

国内用习惯还是oschina的git.下面介绍下我的做法

  1. 在oschina的git下建立一个新的repository ，名叫blog（与hexo文件夹名一样即可）。

  2. 在本地进入blog文件夹，用命令git init创建仓库。

  3. 设置远程仓库地址，并更新

    git remote add origin git@xxx //你的git远程地址
git pull origin

  4. 修改.gitignore文件（如果没有，请创建），在里面加入public/ .deploy/ themes/，这些是每次generate和deploy都会更新，对我们没用，因此忽略这两个文件的更新。

  5. 使用命令 git add .，将所有文件提交到缓存区。

  6. 使用命令 git commit -m “add all files” ，将这些文件提交到本地仓库。

  7. 使用命令git push origin master，将本地仓库的改动推送到远程git仓库。

现在在任何一台电脑，只需要git clone,就可以将hexo的源文件复制到本地了。
之后，当写博客后，只需要git add .再git commit -m再git push即可提交到远程仓库。
当远程仓库有更新时，使用git pull或者git fetch就可以同步代码到本地了。

为什么千万不要重写代码？

發表於 2015-07-27   |   分類於 思路想法   |  

转载自: 地址
微信看到的一篇文章,比较认同

程序员都有一颗工程师的心，所以当他们到一片新的场地想做的第一件事就是，将旧的一切推倒重来。是的，他们决不会满足于简单的增量劳动。

或许这种微妙的心理定位可以解释：为什么程序员进入新项目组后宁愿丢掉旧代码重新写，也不愿意修修补补。他们认为旧代码简直一团糟。

但是，事实上真是这样吗？你之所以认为旧代码一团糟，其实是由编程的一个基本定律决定的，那就是：写代码容易，读代码难。

为什么你觉得旧代码异常混乱？因为读代码更难。

这大概就是代码Reuse难以实现的原因。 这就是你组里的每个人都喜欢用不同的功能将分割的字符串转换成一个数组。比起猜测旧的功能是怎样实现的，重新写一个自己的功能要简单和有趣多了。

作为这个公理的推论，你可以问问身边的程序员他们正在奋战的代码怎么样？「简直是一塌糊涂！」他们肯定会这样说。「我简直想推倒重来！」

为什么认为代码这么糟糕呢？「额，看看这个功能，竟然有两页长！完全不知道这些东西为什么在这里！完全不知道这些API是干什么的。」他们会这样回答你。

（漫画：读别人的代码是一种怎样的体验？）

曾经，Borland的创始人 Philippe Kahn当初就是向记者们吹嘘：Quattro Pro会比Microsoft Excel要好用得多，因为它是从头开始编写的，全部都是新的源代码！

但是，认为新代码比旧代码好简直就是荒谬。旧代码是已经运行过的，测试过的。无数的bug在被发现前都上线运行过，发现之后程序员们可能在花了好些日子才修复了这些bug。这种修复可能是一行代码，也可能是几个字符，无数的时间和精力都花在了这些bug修复上。

当你决定抛弃这些旧代码从零开始的时候，你也丢掉全部前任努力的结果。

新代码一定比旧代买好？NO，重写可能会带来更大的风险。

对技术领导者来说，重写项目的代码也是一个异常艰难的决定。因为从公司层面说，重现代码甚至会威胁产品的市场竞争力。一旦决定重写代码，那么与竞品相比，你可能落后了2~3年——在软件行业，这时间可够长的。

你理想中的新代码会带来产品功能的提升▼

但事实上，即便重写的新代码可以实现旧代码的所有功能和需求，但是为产品带来的市场竞争力只有边际提升。因为重写用的新技术、新语言、新框架并没有给产品带来质的飞跃。

更不用说在重写的漫长过程中可能会遇到一些意外情况，比如：

1、缺钱：资金链的断裂▼

2、缺人：核心程序员离职

最终导致效果不佳：达不到原产品应有的所有功能和需求，白白浪费了时间和金钱，也丢掉了市场竞争力。▼

所以重写代码意味着，你在把自己置身于非常危险的境地，可能几年后你也写不出比以前更好的代码。你只是花了一大笔钱把已经存在的代码又写了一遍。

当你觉得眼前的旧代码很烂时，该怎么办？

你觉得旧代码写的很烂，那又怎样呢？它们已经上线，已经在实际运行中经受住了考验。所以当你发现前任留下的代码乱七八糟的时候，不妨冷静下来，从以下三个方面入手理解代码、改善代码：

1、代码的结构有问题

如果一段网络代码突然弹出了自己的对话框，应该是UI代码需要被处理。这些问题可以被解决掉，你要一次次小心地移动代码，重构，改变接口。还需要一位细心的工程师立马仔细地检查这些改变是否有问题，从而不打扰到其他人。事实上，甚至比较大的结构变化也可以不扔掉代码来完成。

大牛程序员Joel Spolsky回忆说，曾经在某个项目中，他和他的团队花了好几个月重新架构在一点上：把代码动来动去、清理、创建有意义的基类，并创建了模块之间的完美接口。但是他们始终非常小心翼翼，并没有产生新的bug、也没有丢掉任何旧代码。

2、代码的效率不高

曾经，Netscape的渲染代码被传非常缓慢。但事实上，这只会影响该项目的一小部分，这部分是你可以优化甚至重写的。你完全不必重写全部代码。优化速度的1％工作量，会让你获得99％的爆炸性提高。

3、代码写得很丑

有些代码真的写的很丑，比如Joel曾参与一个项目，开始用下划线做开始的成员变量约定，但后来改用更标准的「M」。所以一半的功能用「」开始，一半用「M」开始，这看起来真的很丑陋。但这个问题5分钟就能解决，而不用从头开始写全部的代码。

最后，你要记住，从头开始再写一遍并不意味着你会写出比以前更好的代码。因为你没有参与到上一个版本的创建，所以你其实根本就不算有经验。一旦你准备推倒重写，你可能会再犯一遍版本一犯过的错，甚至会产生更多的新问题。

100offer说：面对糟糕的旧代码，Keep Calm & Carry On !

在大型商业项目中，推倒重来是非常危险的行为。当然，如果你是在做实验，想到新算法可以随时重写。

如果你跳槽、或刚接手一个新项目，面对看上去异常混乱的旧代码，请冷静下来，忍住推倒重写的冲动，想想上面这些经验之谈。
